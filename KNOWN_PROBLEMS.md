# Known Problems of the Current System (AS‑IS)

*This document is a **brief summary**. For a full description, refer to the [`README.md`](./README.md) and the domain‑specific documents in [`DDD/`](./DDD).*

Despite being a live system with paying customers, the current architecture (Firebase‑first) has reached its limits. The following problems block further scaling and feature development.

---

## 1. Firebase as a Single Point of Failure

- **Source of truth** – Firebase Realtime Database acts as the primary database, event bus, sync engine, and offline cache.
- **Implicit side effects** – Business logic is triggered by database listeners, making it hard to reason about and debug.
- **No real transactions** – Complex workflows (e.g., finishing an inventory session) rely on fragile client‑side coordination.

---

## 2. Unreliable Sync & Offline Logic

- The iPad app stores everything locally and relies on Firebase listeners for changes, but conflicts are resolved arbitrarily.
- There is **no explicit offline‑first API** – the current implementation mixes offline caching with realtime subscriptions, causing inconsistent states.
- Re‑syncing after network loss often requires manual intervention.

---

## 3. No Clear Domain Boundaries

- All logic lives in Firebase triggers, PHP web scripts, and Swift client code – **business rules are scattered**.
- For example, product creation, moderation, and inventory counting all touch the same Firebase nodes, leading to accidental coupling.
- It is almost impossible to change one feature without affecting others.

---

## 4. Limited Scalability

- Firebase Realtime Database has predictable performance limits beyond tens of thousands of inventory items per session.
- The current schema does **not** support multi-organisation or cross‑team aggregations efficiently.
- Reporting (PDF / XLSX) is generated by ad‑hoc queries that time out on larger datasets.

---

## 5. Recipes Cannot Be Used in Inventory

- Recipes exist as structured data, but the legacy inventory flow never broke them down into components.
- Any attempt to enable recipe‑based counting would require rewriting the inventory core – hence it is blocked.

---

## 6. Procurement (Orders) Is External & Manual

- The current “orders” feature is a separate web form that sends emails – **no integration with inventory**.
- There is no supplier management, no order status tracking, and no negotiation history.
- This breaks the operational loop (Inventory → Order → Inventory) that is essential for daily use.

---

## 7. No Historical Integrity Guarantee

- Because Firebase allows direct writes, product or recipe changes can **silently alter past inventory reports**.
- There is no snapshot mechanism – historical data is not immutable.

---

## 8. Moderation Is Fragile

- The moderation queue lives in Firebase, with manual SQL scripts to move approved products to the global catalogue.
- No audit log for moderator actions. No support for “edit” proposals of existing products.

---

## 9. Technology & Maintenance Debt

- **Mixed stacks** – Swift (iOS), PHP (web), MySQL (partial replica), Firebase configuration.
- Adding new features requires changes in three different codebases, often with duplicate logic.
- No automated tests for end‑to‑end inventory workflows.

---

## Conclusion

The current system **works for early adopters**, but it cannot scale to hundreds of teams, support procurement, enable recipe‑based inventory, or guarantee immutable historical data. A **full architectural rewrite** (backend + data layer) is required – preserving business logic and UX – while keeping the system live during migration.
