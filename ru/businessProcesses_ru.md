# Бизнес-процессы — AS-IS → TO-BE

Данный документ описывает реальные бизнес-процессы,
которые система поддерживает **сейчас (AS-IS)**,
и те, которые планируются к реализации **в ближайших фазах (TO-BE)**.

Фокус — на **операционной реальности**, а не на фичах.
Каждый процесс отражает то, как бары, рестораны
и корпоративные клиенты работают на практике.

---

## 1. Онбординг бизнеса

### AS-IS

**Участники**
- Владелец бизнеса / менеджер
- Система

**Процесс**
1. Пользователь регистрируется
2. Пользователь создаёт Организацию
3. Система автоматически:
   - назначает пользователя с максимальными правами в Организации
   - создаёт первую Команду
4. Пользователь:
   - настраивает Команду
   - создаёт Локации
   - приглашает сотрудников

**Ключевые свойства**
- Минимальный порог входа
- Отсутствие предварительной настройки
- Подходит как для одного бара, так и для сетей

**Бизнес-ценность**
- Быстрое начало работы
- Низкая когнитивная нагрузка
- Не требуется помощь sales / support

---

### TO-BE (Расширенный Enterprise)

- Несколько Команд в одной Организации
- Кросс-командная аналитика
- Корпоративные роли и права
- Иерархические организации (будущая фаза)

---

## 2. Управление продуктами и модерация

### AS-IS

**Участники**
- Сотрудники бара
- Менеджеры
- Внутренняя команда модерации

**Процесс**
1. Пользователь ищет продукт в каталоге
2. Если продукт найден:
   - выбирает существующий Product Variant
3. Если продукта нет:
   - пользователь создаёт новый продукт
4. При создании продукта пользователь выбирает категорию:
   - если выбрана стандартная категория (Whisky, Wine, Soft Drink и т.д.):
     - продукт отправляется в очередь модерации
   - если выбрана категория `Other`:
     - продукт становится локальным
     - продукт **не отправляется на модерацию**
     - продукт виден только в рамках текущей команды / организации
5. Модератор проверяет продукт:
   - может изменить **любые поля**
     (label, variant, barcode, category, атрибуты)
   - может:
     - принять продукт  
       (записать в глобальный каталог, после чего продукт становится
       доступным всем пользователям)
     - присвоить продукт к существующему  
       (`assign to` по ID существующего продукта)
     - отклонить продукт (нажать `ignore`)
6. При `ignore`:
   - продукт автоматически переводится
     в категорию личных продуктов пользователя
     под категорией `New Product`,
     если пользователь не задал другую категорию вручную
   - продукт становится локальным
   - продукт не попадает в глобальный каталог

**Бизнес-ценность**
- Масштабируемая продуктовая база
- Контроль качества данных
- Основа для аналитики и интеграций

---

### TO-BE (Future development)

- Кабинеты производителей и поставщиков
- Каталоги от брендов
- Внешние процессы модерации

---

## 3. Инвентаризация (CORE FLOW)

### AS-IS

**Участники**
- Инвентаризационный персонал
- Менеджеры
- Система

**Процесс**
1. Пользователь открывает Inventory Session
2. Система:
   - работает оффлайн (если нет соединения с интернетом)
   - поддерживает BLE-весы
3. Пользователь считает:
   - Product Variants поштучно или остатки (остатки в бутылках или вес личных продуктов, таких как чай, кофе и так далее)
4. Пользователь завершает сессию
5. Система:
   - замораживает снапшот
   - считает расход и расхождения
6. Результаты доступны для:
   - отчётов
   - экспорта
   - аналитики

**Ключевые свойства**
- Offline-first
- Snapshot-based
- Неизменяемая история

**Бизнес-ценность**
- Экономия времени
- Снижение потерь
- Прозрачный склад
- Основной платный функционал

---

### TO-BE

- Рецепты
- Агрегация по локациям
- Организационный уровень инвентаря
- Автоматические рекомендации по дозакупке

---

## 4. Рецепты и преби́чи

### AS-IS

**Участники**
- Бармены
- Менеджеры

**Процесс**
1. Пользователь создаёт Рецепт
2. Рецепт состоит из:
   - структурированных Product Variants
   - фиксированных количеств
3. Рецепт версионируется
4. Во время инвентаризации:
   - пользователь считает готовый объём
   - система разбивает его на компоненты
5. Рецепт не участвует в инвентуре (ошибки первой версии)

**Ключевые свойства**
- Рецепты — это шаблоны
- Инвентаризация ссылается на версии рецептов
- Breakdown сохраняется в снапшоте

**Бизнес-ценность**
- Корректный учёт остатков
- Реалистичная аналитика расхода
- Поддержка реальных барных процессов

---

### TO-BE

- Учёт рецептов в инвентуре:
   - пользователь считает готовый объём
   - система разбивает его на компоненты
- Шеринг рецептов между командами
- Атрибуция авторства
- Creator economy (отдельная фаза)

---

## 5. Заказы и закупки (Procurement)

### AS-IS (Наследие / частично)

**Участники**
- Менеджер, бармен
- Поставщик
- Система

**Процесс**
1. Пользователь вручную определяет нехватку товара
2. Создаёт заказ
3. Отправляет заказ по email
4. Поставщик отвечает вручную

**Ограничения**
- Не связано с инвентаризацией (существует сейчас отдельным приложением на телефон)
- Нет статусов поставщиков
- Ручная сверка

---

### TO-BE (Планируется, высокий приоритет)

**Участники**
- Менеджер
- Поставщик (email / web)
- Система

**Процесс**
1. Завершается инвентаризация
2. Система определяет недостающий товар
3. Пользователь создаёт заказ прямо из данных инвентаря
4. Заказ отправляется поставщику по email
5. Поставщик:
   - подтверждает
   - комментирует
   - или отклоняет
6. Статус заказа обновляется в системе
7. Может использоваться и без инвентуры

**Жизненный цикл поставщика**
- inactive → pending → accepted / rejected

**Бизнес-ценность**
- Замкнутый цикл:
  Инвентарь → Заказ → Инвентарь
- Высокая ежедневная полезность
- Бесплатный входной функционал
- Основа для B2B-интеграций

---

## 6. Управление поставщиками

### AS-IS / TO-BE (гибрид)

**Участники**
- Менеджер
- Поставщик
- Система

**Процесс**
1. Пользователь создаёт профиль поставщика
2. Поставщик по умолчанию неактивен
3. Пользователь отправляет pairing request
4. Система:
   - валидирует email
   - избегает no-reply адресов
5. Поставщик:
   - принимает → активен
   - отклоняет → rejected

**Ключевые свойства**
- Поставщика не заставляют использовать софт
- Email-first взаимодействие
- Проверка «живого контакта»

**Бизнес-ценность**
- Низкий порог для поставщиков
- Более высокий процент ответов
- Надёжность заказов

---

## 7. Аналитика и отчётность

### AS-IS

**Для клиентов**
- Аналитика отсутствует

**Для платформы**
- Список команд (юзеры)
- Список используемых продуктов по командам
- Логи (только с планшета)

---

### TO-BE

- Логи пользования дэшбордом
- Аналитика себестоимости
- Учёт списаний и waste
- Предиктивные заказы
- Бенчмаркинг (анонимизированный)

---

## 8. Модерация (сквозной процесс)

### AS-IS

**Область**
- Продукты
- Атрибуты
- Бренды

**Процесс**
- Очереди
- Правила
- Human-in-the-loop

**Бизнес-ценность**
- Качество данных
- Готовность к маркетплейсу
- Регуляторная безопасность

---

## Ключевые принципы процессов

> Инвентаризация — источник истины  
> История неизменяема  
> Заказы замыкают операционный цикл  
> Пользователи действуют, сущности владеют  
> Монетизация не должна ломать операционку  

Документ намеренно не описывает UI
и фокусируется исключительно
на реальных операционных процессах.
