## 0. Что это за система

**Getti** — это система инвентаризации для **HoReCa**, в первую очередь для баров и ресторанов.

Система решает реальные операционные задачи:
- быстрая и точная инвентаризация,
- минимизация человеческих ошибок,
- офлайн-работа на объекте,
- централизованный каталог продуктов,
- интеграции с POS-системами.

Продукт изначально разрабатывался **Firebase-first** для быстрого запуска MVP
и проверки гипотез.  
Система **работает**, есть **платящие клиенты**, однако текущая архитектура
**плохо масштабируется** и исключает дальнейшее развитие.

**Текущая цель проекта:**  
Постепенно перестроить backend и слой данных, **сохранив бизнес-логику и UX**,
и уйти от legacy-использования Firebase.

## 1. Участники системы

### 1.1 Пользователи (клиенты)

**Типовые роли:**
- **Владелец / менеджер** — управление командой, продуктами, инвентаризациями
- **Бармен / сотрудник** — проведение инвентаризации
- *(в будущем)* бухгалтер / финансовая роль

**Интерфейсы:**
- **Мобильное / iPad-приложение** — инвентаризация (offline-first)
- **Веб-приложение** — управление, финализация, отчёты

---

### 1.2 Модератор (внутренний)

Только сотрудники Getti.

**Задачи:**
- модерация новых продуктов,
- поддержка глобального каталога,
- исправление ошибок (вес, изображения, категории),
- борьба с дублями.

Инструменты модерации **не доступны клиентам**.

---

### 1.3 Разработчик

Задачи разработчика:
- понять предметную область,
- работать с backend и моделями данных,
- **не расширять legacy-логику Firebase**,
- участвовать в постепенной миграции архитектуры.

---

## 2. Текущая архитектура (As-Is)

### 2.1 Общая схема

Исторически система выглядит следующим образом:
Web  → Firebase ← Mobile
↓
SQL (частичная реплика)

Firebase выполняет роль:
- аутентификации,
- базы данных,
- event-шины (listeners),
- синхронизации,
- офлайн-поддержки.

Это было оправдано на этапе MVP,
но сейчас стало архитектурным ограничением.

---

### 2.2 Firebase — текущие функции

В Firebase хранятся:
- пользователи и команды,
- локации (бары),
- продукты (глобальные и локальные),
- очередь модерации,
- инвентаризации,
- поставщики и заказы,
- часть legacy-функциональности.

Логика сильно завязана на:
- realtime listeners,
- неявные сайд-эффекты,
- автоматическую синхронизацию.

---

### 2.3 Веб-приложение (As-Is)

Технологии: HTML / CSS / JS / PHP

Функции:
- регистрация и логин,
- управление командой,
- работа с продуктами,
- финализация инвентаризаций,
- экспорт PDF / XLSX.

Технически:
- напрямую пишет в Firebase,
- частично читает данные из SQL.

---

### 2.4 Мобильное / iPad-приложение (As-Is)

Технологии: Swift

Функции:
- офлайн-инвентаризация,
- интеграция с BLE-весами,
- локальное хранение данных,
- отправка данных через **Save & Send**.

Offline-first — **ключевая фича**, которая должна сохраниться.

---

### 2.5 SQL-база (As-Is)

Технология: MySQL

Используется как:
- частичная нормализация,
- хранилище глобальных справочников.

SQL **не является источником истины**.

## 3. Основные доменные сущности

### 3.1 Продукты

**Глобальный каталог продуктов (GPC):**
- алкоголь,
- безалкогольные напитки,
- табак,
- проходят модерацию,
- общие для всех команд.

**Локальные продукты:**
- специфичные для команды,
- рецепты и пре-батчи,
- прочие позиции,
- не проходят модерацию.

---

### 3.2 Рецепты (Pre-batch)

- рецепт — структурированный список ингредиентов,
- используется в инвентаризации,
- автоматически раскладывается на компоненты.

---

### 3.3 Инвентаризации

Инвентаризация привязана к:
- команде,
- локации,
- времени.

Состоит из:
- взвешенных позиций,
- штучных позиций,
- рецептов.

---

## 4. Модерация продуктов

### Сценарий A — создание продукта
1. Пользователь создаёт продукт.
2. Если продукт не `Other`, он попадает на модерацию.
3. Модератор:
   - **принимает** → добавляет в GPC,
   - **привязывает** → к существующему продукту,
   - **игнорирует** → продукт остаётся локальным.

---

### Сценарий B — редактирование глобального продукта
1. Пользователь редактирует продукт из GPC.
2. Создаётся локальный override.
3. Изменения попадают на модерацию.
4. Модератор принимает решение.

---

## 5. Целевая архитектура (To-Be)

### 5.1 Ключевой сдвиг

❌ Firebase как «мозг системы»  
✅ Backend + SQL как источник истины

Firebase временно остаётся для:
- аутентификации,
- push-уведомлений,
- обратной совместимости.

---

### 5.2 Целевая схема
Mobile / Web
↓
Backend API
↓
PostgreSQL
↓
Async workers (sync, moderation, exports)

Мобильное приложение:
- остаётся offline-first,
- синхронизируется через API,
- не использует Firebase listeners напрямую.

---

### 5.3 Планируемый стек

### Backend (целевая архитектура)

- **Node.js (TypeScript)** — основной backend API
- **PostgreSQL** — основной источник истины (source of truth)
- **Асинхронные воркеры** — синхронизация, модерация, экспорт данных

### Опционально / будущие этапы

- **Python-сервисы** — аналитика, машинное обучение, обработка данных

**База данных**
- PostgreSQL
- явная схема
- миграции

**Инфраструктура**
- Zone.ee
- резервное копирование (3 месяца)
